# AGENTS.md

## 项目简介

个人记账系统，包括 APP 端、WEB 端和 API 服务。
本文件只记录 Agent 执行规则；项目架构、技术栈、目录结构和系统边界详见同目录 `ARCHITECTURE.md`。

## 硬规则

- 修改代码前，先给出可选方案并等待确认。
- 默认使用中文沟通、说明和评审。
- 未经确认，不进行实质性代码写入、删除、迁移、重构。
- 读取文件、运行只读检查、给出方案不需要确认；写入、删除、迁移、重构、安装依赖、提交代码必须确认。
- 不得覆盖、回滚或污染用户已有未提交改动。
- 新发现的稳定约定，需要提示是否沉淀到公共规范。
- 业务源文件原则上不超过 300 行；生成文件、migration、lockfile、快照、配置聚合文件除外。
- 使用结构化日志，禁止 `console.log`

## 基本原则

- 不改变既有功能、API 语义、数据结构或用户可见行为，除非任务明确要求。
- 不随意修改数据库表结构、接口路径、响应结构、金额策略、时间策略和权限边界。
- 项目未上线阶段可以调整不合理的目录或文件路径，但迁移必须保持当前功能行为不变。
- 类型治理服务于架构边界和业务表达，不把“修 TypeScript 类型”当作脱离业务边界的独立主线。
- 优先遵循现有代码风格、目录约定、子项目 `AGENTS.md` 和同级 `ARCHITECTURE.md`，再考虑新增抽象。
- 较大改动完成后，必须说明验证结果、剩余风险和是否仍存在值得继续治理的热点。

## 文件体量与拆分

- 文件行数是健康信号，不是机械 KPI；拆分目标是提升可读性、边界清晰度和可测试性。
- 业务源文件原则上不超过 300 行；如职责单一、阅读路径连续、强行拆分会增加跳转成本，可以合理保留，并在评审或交付说明中解释原因。
- 测试文件、schema、migration、lockfile、快照、生成文件、配置聚合文件可以例外，但仍应按语义分组。
- 合理拆分优先级：
  1. 先拆确实可复用或可单测的纯函数、格式化、映射、校验、状态收敛 helper。
  2. 再拆职责完整的展示组件，例如 table、dialog、toolbar、action bar、empty state。
  3. 再拆 controller hook、query hook、workflow hook、service 或 use case，前提是能减少主流程复杂度。
  4. 最后再考虑跨包抽象，避免提前抽象。
- 禁止为了压低行数制造只透传 props 的组件或函数，除非它表达了稳定业务概念。
- 禁止新增含义模糊的 `helpers2`、`misc`、`common`、`utils` 大杂烩文件。
- 禁止把强相关的连续业务流程拆成多个需要来回跳转、靠大量参数串联的小文件。

## 开发命令

默认使用 pnpm，通过 Turborepo 编排任务。

常用命令：

- `pnpm dev`
- `pnpm build`
- `pnpm test`
- `pnpm lint`
- `pnpm check`

如具体子项目命令不同，以对应 `package.json` scripts 和 `turbo.json` pipeline 为准。

## 测试要求

- 新增业务逻辑必须有单元测试
- 修复 bug 必须先补充能复现问题的测试
- API 关键流程需要集成测试
- 测试不得依赖真实外部服务
- 不允许为了通过测试删除或弱化有效断言

## 实现边界

- 代码组织、目录职责、共享包边界、数据库归属、金额和时间策略详见 `ARCHITECTURE.md`
- 不得为了完成局部任务绕过既有架构边界；需要调整架构时先给方案并等待确认
- 前端不得直接依赖服务端内部实现或数据库层；移动端和 WEB 端只通过服务端 API 访问业务数据。
- 服务端必须作为业务规则、权限边界、事务一致性和数据访问入口。
- `packages/shared` 只放跨端稳定共享的类型、常量、纯工具函数和运行环境无关的校验/格式化逻辑。

## 日志规范

- 禁止使用 `console.log`、`console.warn`、`console.error`
- 服务端必须使用结构化 logger
- 日志中不得输出密码、token、银行卡号等敏感信息
- 错误日志应包含可追踪上下文，但不能泄露用户隐私

## 文档查询

涉及库、框架、SDK、API、CLI 工具或云服务的问题时，必须使用 ctx7 查询当前文档。

## Git 与工作区安全

- 不得执行 `git reset --hard`、`git checkout --`、`git restore` 等可能覆盖用户或其他 agent 改动的命令，除非用户明确要求。
- 不得擅自执行 `git add`、`git commit`、`git push` 或创建 PR；需要提交时先说明范围并等待确认。
- 如果工作区已有未提交改动，必须区分自己的改动和既有改动，不得回滚或格式化无关文件。
- 删除文件、移动目录、批量重构、安装依赖、执行 migration、修改 lockfile 都必须先确认。
- 交付时必须说明修改文件、验证命令和剩余风险；未能执行的验证也要说明原因。

## 验证与交付

- 局部改动优先运行对应子项目或目标文件的检查，再按风险选择全量检查。
- 文档或配置类改动至少检查 diff 是否符合预期；代码改动按影响面选择 `pnpm lint`、`pnpm test`、`pnpm check`、`pnpm build`。
- 修复 bug 必须先补充能复现问题的测试，再实现修复。
- 新增业务逻辑、金额计算、日期处理、权限判断、API 关键流程必须有相应测试。
- 提交或交付前应确认：
  - 相关测试通过，或明确说明未运行原因。
  - 没有新增 `console.log`、`console.warn`、`console.error`。
  - 没有真实密钥、token、密码、连接串等敏感信息。
  - `git diff --check` 无空白错误。

## Agent 协作规范

- 多 agent 并行时必须明确互不重叠的写入范围。
- agent 之间不得回滚、覆盖或格式化彼此改动。
- 主线程负责最终集成、冲突判断、验证收口和剩余风险统计。
- 如果发现可复用的稳定约定，先提示是否沉淀到公共规范，再修改 `AGENTS.md` 或相关架构文档。

## 提交规范

提交信息使用：

- `feat`: 新功能
- `fix`: 修复
- `refactor`: 重构
- `docs`: 文档
- `test`: 测试
- `chore`: 工具链、依赖、配置
- `ci`: CI/CD 配置
- `perf`: 性能优化

提交前应确认：

- `pnpm test` 通过
- `pnpm lint` 通过
- `pnpm check` 通过
- 没有 `console.log`
- 没有真实密钥
